## やったこと

### Validation 実装の進捗共有

- `research-workspace` を使って、validation 実装の整理と外部実装リポジトリでのコード修正を進めた
- `val loss` と tracking 指標ベースの validation を分けて扱う方針で実装を進め、少なくとも smoke test レベルでは動作確認できている
- `train` / `val loss` / tracking validation を同時に記録する形でいったん実装した
- `val loss` は毎 epoch 回る実装になっており、今後運用に合わせて頻度や命名を整理する必要がある

### 実装運用上の論点共有

- `research-workspace` は複数リポジトリをまたいで文脈を残しながら作業できる点が有用だった
- 一方で、`brainstorm` スキルが `specs` / `plans` を十分作る前に実装へ進みやすい挙動があり、運用上の注意点として認識された
- `comet_ml` が仮想環境から消えていたため、実験ログを一部残せなかったが、原因は `uv` 移行時の環境変化の可能性が高い

### MIRU ポスターの相談

- ポスターは「SAM2 / SAMURAI をどう拡張するか」を主題に据える構成で進める
- Mamba と Kalman filter の比較は冒頭ではなく、SAM2 / SAMURAI 改善のための方法として後段に配置する
- 説明図は sliding window と state carry を対等に並べるより、既存法の流れが分かる図と state carry の要点に絞る方針が示された
- 結果表だけでなく、オクルージョン時の定性的なトラッキング可視化を用意することが宿題になった

## 話したこと

### validation 命名と役割整理

- `tracking-val` という表現では、何を評価しているのかが第三者に伝わりにくいという指摘があった
- 1-step の `val loss` と、tracking 全体を走らせて HOTA などを見る評価を明確に区別できる名前に直す必要がある
- 当面は `train` / `val loss` / tracking 全体評価の関係を観察するために、すべて記録する方針で進める

### TrackEval 統合の切り分け

- 現状の「tracking validation が重い」という問題は、評価そのものが重いのか、統合過程で余計な処理が入ったのかを切り分ける必要がある
- まずは TrackEval 側で、既存の快適に動いていた導線を関数化し、その関数単体で正しく動くことを確認してから学習コードへの組み込みを評価するべき、という方針になった
- 「関数化」と「学習ループへの統合」という二つの変更を同時に入れているため、要因分離が不十分だという整理がなされた

### 計算時間増加の原因調査

- tracking validation の計算時間増加は、単に「重い」と言うのではなく、どの処理が追加コストを生んでいるのかを調べるべきという指摘があった
- 特に、想定外に計算グラフへ入っている処理や、統合時に増えたオーバーヘッドの有無を確認する必要がある
- PyTorch Lightning の profiler を含め、適切な profiler を使ってボトルネックを特定する方向で進める

### MIRU ポスターの構成

- ポスター上部は「SAM2 / SAMURAI の改善」が主題であり、Mamba はその解決手段として位置づける
- 中段では state carry の説明を最小限にし、必要なら既存法の sliding window 的な扱いを分かりやすい図で触れる
- 本番に向けて、定量表に加えてトラッキング可視化や動画提示も準備することが望ましいという話になった

## 決定事項

- validation 系の名称を見直し、`val loss` と tracking 全体評価の役割差が伝わる表現に整理する
- TrackEval 統合の前に、TrackEval 側で関数化した導線が単体で正しく動くことを確認する
- tracking validation の計算時間増加は profiler を使って原因を切り分ける
- MIRU ポスターは「SAM2 / SAMURAI の改善」を主軸に構成し、Mamba は方法として説明する
