## やったこと

### 学習方法の比較分析

- 前回までTrackSSMとMambaTrackの比較を行っていたが、lossの下がり方に大きな差がある原因を分析した
- **TrackSSMのみ入出力形式が異なることを発見**
  - MambaTrack：bboxのdeltaを入力し、deltaを出力
  - TrackSSM：bboxとdeltaを入力し、bboxを出力
  - これが前回までのloss比較の違いの主因だった

- MambaTrackとState carry型の比較
  - MambaTrackは25エポック（デフォルト設定のまま停止）、State carryは100エポック学習
  - lossはMambaTrackより State carry型の方が安定して下がっていた
  - MambaTrack自体まだ学習が不十分な可能性が高い（25エポックでは足りない）

### TrackEvalによる評価結果

- MambaTrack・TrackSSMのHOTA：約0.33〜0.34
  - 本来の目標（SAM2系で約0.56相当）と比較してかなり低い
  - 25エポック学習不足 + TrackSSMは実験設定が怪しい可能性
  - スクリーンショットの貼り間違いもあり、CometMLのリンクで共有・確認する方針に
- State carry型：そこそこの性能が出ており、100エポックで収束しかけている
  - CometML: https://www.comet.com/sun0801/mamba-mot/46dfef23fb9c4f4fbba691bac8423e82

### 学習率スケジューラの問題

- State carry型のLR schedulerがほぼ固定になっていることが発覚
  - LRグラフがほぼ横一線（一切動いていない）
  - Warmup → Exponential decayのようなスケジューラを設計すべきだが、現状warmupが急激すぎ・decayも急激すぎる
  - **lossが一瞬で下がってその後ほぼ変化なし** → 過学習の可能性が高い
  - ハイパラ（LR schedule含む）を何も調整していない状態。まずvalidationを入れて過学習が起きているか確認する

### Mambaの内部状態汚染（Hidden State Contamination）の調査

- MOTでsliding window型が多く、state carry型がほぼ使われていない理由の仮説として、hidden state contaminationを調べた
- 参照文献：
  - *Hidden State Poisoning Attacks against Mamba-based Language Models*（Mambaの脆弱性研究）
  - *SPM-Track: A State-Persistent Mamba Framework with Hierarchical Context Management for Lightweight Visual Tracking*（状態の信頼度を監視してstate carryするUAV追跡）
- 内部状態の汚染を可視化した論文はまだ見当たらない

## 話したこと

### Validationの設計について

**論点**：MOTにおけるvalidationは一般的にどう行われるか？

整理された理解：
- **学習中のval loss**（本来の意味でないval）：次のbboxを当てるタスクで、過学習が起きていないかモニタリングするためのもの。性能に直結しない
- **TrackEval等による最終評価**：学習が全部終わった後にvalidationデータでトラッキングを実行し、HOTAなどを出すのが一般的

**計算コストの議論**：
- val用動画は約30本
- 1回の評価時間：約5分
- 1エポックの学習時間：約10〜20分
- → 毎エポックvalを入れると1/3以上が評価に取られる
- → **5エポックごとの評価であれば、全体に占める比率が低くHOTA等の最終評価指標を最初から見ることも現実的**

**決定**：
- まずは「本来の意味ではないval（過学習チェック用）」を実装する
- 本来のHOTA評価（TrackEval）も5エポックごとに入れることを検討する
- 複数GPU対応を見据える場合、Lightning等のプロセス間通信機能を正しく使えるかが実装上の鍵

> コード上の関数名・ログ名は `val` でよいが、「datasetとしてのvalidation set」と「最終評価指標としてのvalidation」を混同しないこと

### TrackSSMの扱いについて

- 非公式実装のため実装が怪しい可能性が高い
- 実験設定（入力形式、エポック数、LRスケジュール）が揃っていない状態での比較は意味がない
- **公平な比較のためには、条件統一が先決**

### Mambaの内部状態汚染の可視化アイデア

- 汚染の**定義・定量化**ができることが前提
  - 汚染度合いを数値で表せれば「トラッキング中に汚染が上昇した」という主張ができる
- アイデア：
  - 内部状態（行列）をPCA等で軌跡プロット → 正常トラックはきれいな軌跡、汚染トラックはぐちゃぐちゃになるか？
  - シミュレーションで同一動画に強制的にオクルージョンを発生させて比較
- **注意点**：MambaのhiddenstateがPCAで意味のある構造として分解できる保証はない（非線形のため）。まず理論面を調べてからやってみることが先決
- *Hidden State Poisoning*論文はAdversarial attackの文脈だが、オクルージョンによる汚染の参考にはなる（種類は違うが現象として類似）
